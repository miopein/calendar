# Roadmap: Console Work Time Tracker -> Android

Цель:
Легковесный трекер рабочего времени, который сначала работает в консоли, а затем переносится в Android-приложение без потери данных.

Схема фаз:

[Фаза 1: Требования и правила данных]
        ->
[Фаза 2: Архитектура хранения (SQLite, миграции, бэкапы)]
        ->
[Фаза 3: CLI MVP (start/stop/status/list/edit)]
        ->
[Фаза 4: Календарь и отчеты]
        ->
[Фаза 5: Надежность и долговечность (тесты, таймзоны, производительность)]
        ->
[Фаза 6: Переход в Android (Kotlin + Room + Compose)]

---

## Фаза 1. Требования и правила данных

Цель:
Зафиксировать, как система должна работать, чтобы не переделывать логику на следующих этапах.

Что делаем:
1. Описываем пользовательские сценарии: start, stop, просмотр, редактирование.
2. Фиксируем ограничения: одна активная сессия, проверка интервалов, правила ошибок.
3. Определяем обязательные поля записи и формат времени.

Результат фазы:
1. Документ с требованиями и валидациями.
2. Базовая схема хранения записей.

Критерий завершения:
1. Все ключевые сценарии описаны.
2. Нет противоречий в правилах данных.

## Фаза 2. Архитектура хранения (SQLite, миграции, бэкапы)

Статус: Done

Цель:
Обеспечить надежное хранение данных на горизонте 10+ лет.

Что делаем:
1. Утверждаем структуру таблиц и индексов.
2. Добавляем версионирование схемы и миграции.
3. Проектируем резервное копирование и восстановление.
4. Проверяем производительность запросов на большом объеме данных.

Результат фазы:
1. Рабочая БД с индексами.
2. Политика бэкапов и инструкция восстановления.

Критерий завершения:
1. БД поднимается автоматически.
2. Восстановление из копии проходит успешно.

Артефакты реализации:
1. Миграции: db/migrations/0001_initial.sql, db/migrations/0002_triggers.sql
2. Управление БД: scripts/manage_db.py
3. Документация фазы: docs/PHASE_2_STORAGE.md

## Фаза 3. CLI MVP (start/stop/status/list/edit)

Статус: Done

Цель:
Сделать минимально полезную консольную версию для ежедневной работы.

Что делаем:
1. Реализуем команды start, stop, status.
2. Реализуем list по дню и месяцу.
3. Добавляем edit для текущих и старых записей.
4. Добавляем валидации и понятные сообщения об ошибках.

Результат фазы:
1. Пользователь может вести учет времени полностью из консоли.
2. Редактирование работает для прошлых периодов.

Критерий завершения:
1. Основные команды работают стабильно.
2. Ошибки пользователя обрабатываются корректно.

Артефакты реализации:
1. CLI: scripts/time_tracker.py
2. Документация: docs/PHASE_3_CLI_MVP.md

## Фаза 4. Календарь и отчеты

Статус: Done

Цель:
Сделать удобную навигацию по истории и аналитику по времени.

Что делаем:
1. Добавляем календарный вид (день, неделя, месяц).
2. Добавляем summary по диапазону дат.
3. Добавляем экспорт в CSV/JSON.
4. Добавляем фильтрацию по проекту или заметке (опционально).

Результат фазы:
1. Быстрый обзор нагрузки по датам.
2. Готовые выгрузки для отчетности.

Критерий завершения:
1. Можно получить часы за любой период, включая давние записи.
2. Экспорт открывается без ошибок.

Артефакты реализации:
1. CLI: scripts/time_tracker.py (calendar, summary, export)
2. Документация: docs/PHASE_4_CALENDAR_REPORTS.md

## Фаза 5. Надежность и долговечность

Цель:
Подготовить систему к долгой эксплуатации без потери данных.

Что делаем:
1. Покрываем критичные сценарии тестами.
2. Проверяем пограничные случаи времени (таймзоны, переходы дат).
3. Добавляем журнал исправлений и защиту от повреждения данных.
4. Проводим тест на большом объеме сессий.

Результат фазы:
1. Стабильная версия CLI для постоянного использования.
2. Подтвержденная устойчивость на больших объемах.

Критерий завершения:
1. Критичные тесты проходят.
2. Нет блокирующих проблем с корректностью времени.

## Фаза 6. Переход в Android (Kotlin + Room + Compose)

Цель:
Перенести продукт на телефон с сохранением данных и бизнес-логики.

Что делаем:
1. Создаем Android-проект и переносим модель данных в Room.
2. Реализуем экраны: Start/Stop, календарь, список, редактирование.
3. Делаем импорт из консольной версии (SQLite/JSON).
4. Проводим функциональное сравнение с CLI-версией.

Результат фазы:
1. Мобильное приложение с теми же ключевыми возможностями.
2. Миграция данных из консольной версии без ручного ввода.

Критерий завершения:
1. Пользователь может продолжить учет на Android с уже накопленной историей.
2. Основные сценарии работают без регрессий.

---

## Предлагаемая очередность запуска работ

1. Сразу довести Фазу 1 до Done: утвердить формат времени и правила бэкапа.
2. Перейти к Фазе 2 и зафиксировать миграции.
3. Запустить Фазу 3 и получить рабочий CLI MVP.

Ключевые решения:
- Язык для консольного этапа: Python
- Хранение данных: SQLite (локальный файл)
- Приоритеты: простота, редактирование записей, долгий срок хранения

Критерии готовности проекта:
- Можно начать и завершить рабочую сессию одной командой
- Можно редактировать текущие и старые записи
- Есть календарный просмотр и отчеты по диапазону дат
- Данные корректно хранятся годами и восстанавливаются из бэкапа
