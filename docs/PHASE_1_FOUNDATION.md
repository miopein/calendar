## 1) Функциональные требования (зафиксировано)

1. Пользователь запускает учет времени (start) и система сохраняет точное время начала.
2. Пользователь завершает учет времени (stop) и система сохраняет точное время окончания.
3. Система не допускает более одной активной сессии одновременно.
4. Пользователь может просматривать записи по дню, месяцу и диапазону дат.
5. Пользователь может редактировать как текущие, так и прошлые записи.
6. Система должна хранить историю за много лет (включая 10+ лет).

## 2) Сценарии использования (зафиксировано)

1. Старт дня:
   - Пользователь вводит команду start.
   - Создается новая сессия с заполненным start_time и пустым end_time.

2. Конец дня:
   - Пользователь вводит команду stop.
   - Для активной сессии заполняется end_time.

3. Исправление записи:
   - Пользователь выбирает запись по id.
   - Меняет start_time и/или end_time.
   - Система валидирует интервалы и сохраняет изменение.

4. Просмотр старых периодов:
   - Пользователь запрашивает период (например, месяц 10 лет назад).
   - Система показывает список сессий и суммарные часы.

## 3) Правила валидации (зафиксировано)

1. end_time не может быть раньше start_time.
2. Нельзя создать вторую активную сессию, если первая не завершена.
3. При ручном редактировании обязательна повторная проверка интервала.
4. Любая правка должна обновлять updated_at.

## 4) Нефункциональные требования (зафиксировано)

1. Легковесность: локальный запуск без серверов.
2. Надежность: хранение в SQLite, периодический бэкап.
3. Расширяемость: структура данных должна быть совместима с Android-версией.

## 6) Критерий завершения фазы 1

Фаза 1 считается завершенной, когда:
1. Требования не противоречат друг другу.
2. Правила валидации покрывают старт/стоп и редактирование.
3. Схема БД поддерживает долгий горизонт хранения и будущие отчеты.
