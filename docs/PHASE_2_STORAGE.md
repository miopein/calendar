# Фаза 2: Архитектура хранения

Статус: Done

## Что реализовано

1. Введены SQL-миграции в папке db/migrations.
2. Добавлено версионирование схемы через таблицу schema_migrations.
3. Добавлен скрипт управления БД: scripts/manage_db.py.
4. Добавлена политика резервного копирования и восстановления.
5. Добавлен скрипт проверки производительности: scripts/benchmark_storage.py.

## Решения по данным и времени

1. Формат времени для приложения: UTC в ISO 8601 на уровне приложения.
2. В SQLite время хранится в TEXT, сравнение интервалов работает лексикографически при едином формате.
3. Для консоли отображение может конвертироваться в локальное время пользователя, но хранение остается в UTC.

## Состав миграций

1. 0001_initial.sql:
   - Таблицы sessions и session_edits.
   - Индексы для фильтров по датам и активной сессии.
2. 0002_triggers.sql:
   - Триггер автообновления updated_at при изменениях записи.

## DB-операции (рабочие команды)

1. Инициализация БД:
   - python3 scripts/manage_db.py init
2. Применение миграций:
   - python3 scripts/manage_db.py migrate
3. Статус миграций:
   - python3 scripts/manage_db.py status
4. Резервная копия:
   - python3 scripts/manage_db.py backup
5. Восстановление:
   - python3 scripts/manage_db.py restore --backup-file backups/<file>.db
6. Проверка целостности:
   - python3 scripts/manage_db.py integrity-check

## Проверка производительности

1. Запуск синтетического теста на большом объеме данных:
   - python3 scripts/benchmark_storage.py --rows 100000 --years 15
2. Что проверяется:
   - скорость массовой вставки
   - диапазонная выборка (примерно 10 лет назад)
   - месячная агрегация часов
   - поиск активной сессии по индексу
3. Минимальное условие успешности:
   - integrity_check возвращает ok
4. Результаты последнего прогона:
   - docs/PHASE_2_BENCHMARK_RESULTS.md

## Политика резервного копирования

1. Частота:
   - Минимум 1 раз в день в рабочие дни.
   - Дополнительно перед крупными правками или массовым редактированием.
2. Хранение:
   - Дневные копии: 14 дней.
   - Недельные копии: 8 недель.
   - Месячные копии: 12 месяцев.
3. Правило безопасности:
   - Перед restore текущая БД сохраняется в файл .pre_restore.
4. Контроль:
   - После restore обязательно запускать integrity-check.

## Критерии завершения фазы 2

1. БД создается автоматически и миграции применяются без ручного SQL.
2. Есть подтвержденный сценарий backup/restore.
3. Есть проверка целостности БД.
4. Есть воспроизводимая проверка производительности на большом объеме.
